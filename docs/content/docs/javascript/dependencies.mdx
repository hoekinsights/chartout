title: Dependencies

The `chartout` npm package ships several entry points. Which ones you import depends on your app stack. See [Integrations](/docs/javascript/integrations) for framework-specific examples.

## Package exports

| Export | Purpose |
|--------|---------|
| `chartout` | `createChartoutModel`, state types, `Theme`, `themeToStyle` |
| `chartout/store` | `openWithViz`, `openWithItem`, `openWithCart`, `svgToBytes`, `products` |
| `chartout/react` | `ChartoutWidget` for React apps |
| `chartout/mount` | `mount()` for vanilla JS, Vue, Svelte, and other non-React hosts |

## Embedding options

**`chartout/react`.** For React 19+ apps. Install `react` and `react-dom` as peer dependencies. `ChartoutWidget` uses the same React instance as your app.

**`chartout/mount`.** For vanilla HTML, Vue, Svelte, and other hosts. Call `mount()` on a container element. The package is self-contained; no extra UI dependencies required. The mount bundle is larger than `chartout/react` because it includes everything needed to render the store.

## Peer dependencies

```json
"peerDependencies": {
  "react": ">=19",
  "react-dom": ">=19"
}
```

Both are **optional**. Install them only when you use `chartout/react`:

```bash
npm install chartout react react-dom
```

If you only use `chartout/mount`, skip React entirely.

## What the widget includes

The store is a **browser-only** UI. It needs a DOM and does not run on a server.

`chartout/mount` and `chartout/react` ship a prebuilt widget bundle. You do not install its internals separately. The bundle already includes:

- **Three.js** for the 3D product preview
- **skia-canvas** for compositing chart PNGs onto the product mockup and updating the 3D texture
- **anywidget** utilities and other UI code

You do not add these to your `package.json`. If you use a meta-framework with server-side rendering, a server build may fail on `skia-canvas` even though the widget only runs in the browser. Import chartout in client code only; see [SSR and meta-frameworks](#ssr-and-meta-frameworks) if the build still errors.

### svgToBytes

`svgToBytes` in `chartout/store` rasterises an SVG chart to PNG bytes using the browser's `OffscreenCanvas` API. That produces the print content for cart placements.

Call it yourself when building `CartItem[]` for `openWithCart`. For `openWithViz` and `openWithItem`, pass an SVG and those helpers call `svgToBytes` internally.

It lives in the slim `chartout/store` bundle, separate from the widget UI that handles the 3D preview and placement interactions.

### SSR and meta-frameworks

If a server build pulls in `chartout/mount` or `chartout/react`, it may resolve `skia-canvas` incorrectly and fail with errors about `.node` files.

Import chartout only in client code (`'use client'` in Next.js, `onMount` in Svelte, client-only components in Nuxt). If the server build still mentions `skia-canvas`, exclude it from the server bundle:

```ts
// vite.config.ts
export default defineConfig({
  ssr: {
    external: ['skia-canvas'],
  },
})
```

```ts
// next.config.ts
webpack: (config, { isServer }) => {
  if (isServer) {
    config.externals = [...(Array.isArray(config.externals) ? config.externals : [config.externals].filter(Boolean)), 'skia-canvas']
  }
  return config
},
```

## Browser APIs

| API | Used by |
|-----|---------|
| `OffscreenCanvas` | `svgToBytes` |
| `fetch` | `products()`, widget catalogue load |
| WebGL | 3D product preview |

Target modern evergreen browsers. Ensure chart SVGs have readable `width`/`height` or a `viewBox` before calling `svgToBytes`.
